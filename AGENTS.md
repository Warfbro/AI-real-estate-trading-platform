# AGENTS.md

## 0. 文件定位
本文件是本项目的最高执行规则。
Codex 在本仓库内执行任何分析、编码、重构、修复、联调、部署脚本编写、文档更新前，都必须先遵守本文件。
如本文件与普通说明冲突，以本文件优先。
如本文件与正式开发档案冲突，先报告冲突，不得自行猜测执行。

---

## 1. 项目总目标
本项目为：微信小程序房产一站式决策与推进平台。
目标是在既有开发档案约束下，持续完成整个项目开发，而不是做一次性片段代码输出。

本项目默认原则：
1. 严格按开发档案推进。
2. 严格按任务顺序推进。
3. 每次只完成一个最小可提交单元。
4. 优先可运行、可验证、可回退。
5. 不擅自扩大范围，不擅自换技术栈，不擅自重构未授权模块。

## 3. 技术与平台硬约束
本项目默认技术边界如下：
1. 客户端：微信小程序
2. 后端能力：微信云开发 / 腾讯云 CloudBase 体系
3. 云函数、云数据库、云存储相关资源：优先通过代码、配置、脚本、项目文档统一管理
4. 不允许把控制台手工操作作为唯一真源
5. 若新增集合、索引、云函数、权限、安全规则、环境配置，必须同时更新代码与文档
6. 若文档已有技术选型，禁止私自替换
7. 若引入新依赖，必须说明用途、影响范围、回退方式

CloudBase CLI 官方用于快速部署项目和管理云开发资源，因此默认优先采用 CloudBase CLI 或代码化方式管理云资源，不把手工点选当唯一方案。:contentReference[oaicite:3]{index=3}

## 4. 默认执行顺序
每次任务都必须按以下顺序执行：

第一步：读取开发档案
第二步：根据process.txt判断当前项目阶段（如果process.txt不存在则重建完整process.txt，文档格式在/process/格式.txt）
第三步：识别本次唯一最小任务
第四步：检查前置依赖是否满足
第五步：输出执行计划
第六步：再开始修改代码
第七步：本地自检
第八步：更新相关文档、埋点、状态
第九步：输出交接总结

禁止行为：
1. 不读文档直接写代码
2. 一次做多个不相干任务
3. 跳过前置依赖直接实现后续模块
4. 顺手改动与当前任务无关的模块
5. 把“建议”当成“已确认需求”

## 6. 人工确认触发标准（必须严格减少无效确认）
默认原则：
只要文档足够明确，Codex 必须直接执行，不要为了小事频繁请求人工确认。

### 6.1 以下情况不需要人工确认，必须直接做
1. 文档中已明确的页面、字段、状态、接口、流程实现
2. 文档中已有模块的补全、修复、联调
3. 样式、命名、目录在现有风格内的小幅调整
4. 为了让当前任务可运行而增加的必要辅助代码
5. 已有接口的参数校验、错误处理、日志补全
6. 已明确埋点的接入
7. 已明确字段的表单、列表、详情、状态展示
8. 已明确规则的代码实现
9. 不改变外部契约的小范围重构
10. 文档中已写明的云函数、数据库、集合、索引、查询、状态流转实现

### 6.2 以下情况才需要人工确认
1. 8 份开发档案之间出现冲突，且无法按优先级自动裁决
2. 需要新增文档中不存在的核心页面、核心表、核心接口、核心状态
3. 需要删除已有能力或废弃既有流程
4. 需要修改外部接口契约、数据库主结构、关键状态机
5. 需要引入新的收费服务、外部账号、密钥、第三方平台权限
6. 需要执行高破坏性操作，例如批量删除文件、覆盖迁移、重置数据、替换大量目录
7. 需要对生产/正式云环境进行不可逆变更
8. 需要在多个可行方案中进行产品级取舍，而文档未定
9. 需要处理法律、支付、隐私、权限边界等高风险问题
10. 明确缺少必需输入，继续执行会大概率返工

### 6.3 若需要人工确认，输出格式必须固定
必须只输出以下 4 项：
1. 触发确认的具体原因
2. 受影响文件/模块
3. 可选方案 A/B/C
4. 若无人确认，推荐默认方案

禁止只说“需要你确认一下”而不给选项。

## 8. 现代化开发工具与工程纪律（必须执行）
为了保证可回退、可定位、可持续开发，必须遵循以下工程纪律：

### 8.1 Git 强制使用
1. 本项目必须在 Git 仓库内开发
2. 每个最小任务必须形成清晰提交边界
3. 未经确认不得把多个不相干任务混在同一次提交中
4. 任务完成后必须给出建议提交信息
5. 禁止在脏工作区下大规模改动

Git 是分布式版本控制系统；`git worktree` 支持同一仓库挂多个工作树并行开发，`git restore` 支持从指定提交、分支或标签恢复文件。:contentReference[oaicite:4]{index=4}

### 8.2 分支与隔离规范
默认建议：
1. main：稳定主线
2. feat/*：功能开发
3. fix/*：缺陷修复
4. refactor/*：局部重构
5. chore/*：工程维护

若需要并行试验、保留稳定版本、同时处理多个任务，优先建议使用 `git worktree` 隔离工作区，而不是直接在一个目录混改。:contentReference[oaicite:5]{index=5}

### 8.3 回退规范
每次任务都必须可回退：
1. 改动前先确认影响文件
2. 重大改动前先给出回退点建议
3. 若改动失败，优先回退到任务开始前的最小稳定状态
4. 任何新增脚本、迁移、配置都要注明回退方式
5. 不允许留下“半改状态且不可恢复”的结果

### 8.4 调试规范
1. 每次任务必须说明如何验证
2. 若是页面任务，至少给出进入路径、操作步骤、预期结果
3. 若是接口任务，至少给出输入、输出、异常分支
4. 若是云函数任务，至少给出调用方式、参数、日志观察点
5. 若是状态流转任务，至少给出前置状态、触发动作、目标状态
6. 若是数据库改动，至少说明受影响对象、写入路径、读取路径

### 8.5 日志与错误定位
1. 新增异步流程、云函数、关键提交动作时，必须加最小必要日志
2. 错误日志必须包含上下文，不得只打印“失败”
3. 不记录敏感信息明文
4. 关键失败必须能从日志回溯到用户动作、接口、对象、阶段

### 8.6 自检规范
每次任务完成后，至少做以下自检：
1. 文档一致性检查
2. 代码编译/语法检查
3. 关键流程冒烟检查
4. 埋点是否遗漏
5. 权限或状态是否破坏既有逻辑
6. 是否存在明显无用代码、死分支、未使用变量

## 9. 数据库、云函数、云资源变更规范
1. 所有云数据库集合、索引、安全规则、云函数、环境配置变更，都必须可追溯
2. 变更时必须同步更新对应文档
3. 若资源变更影响接口或页面，必须同步更新相关调用处
4. 若变更不可逆，必须触发人工确认
5. 若变更可脚本化，优先采用脚本化或配置化方案
6. 不允许先控制台改完再让代码“将就适配”

---

## 10. 编码规则
1. 优先复用现有结构、命名和风格
2. 不为“看起来更优雅”而擅自做大重构
3. 不引入与当前任务无关的基础设施
4. 不创建未被文档支持的新概念
5. 不写只对当前一次性输出有用、但无法持续维护的代码
6. 对不确定逻辑，先查文档，再查现有代码，最后才触发人工确认
7. 生成代码必须以“能继续迭代”为目标，而不是只完成表面功能

---

## 11. 埋点与可观测性规则
1. 所有关键页面和关键动作必须对照 08_埋点清单_关键事件表.txt 实现
2. 如果当前任务涉及新增页面、关键按钮、提交动作、流失点，必须同步补埋点
3. 若文档已有事件定义，必须复用，不得自造新名字
4. 若确需新增事件，必须说明原因，并更新埋点文档或在交接中明确记录

4. 下次继续时先做什么

## 13. 默认工作模式
默认工作模式固定为：

先读 8 份文档
-> 判断当前阶段
-> 选一个最小任务
-> 输出计划
-> 实现
-> 自检
-> 同步文档/埋点/状态
-> 输出交接

除非文档明确要求，否则不得改变此工作模式。
