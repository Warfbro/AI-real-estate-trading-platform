文档名称：验收标准
版本：V1.1（修正版）
适用阶段：首期微信小程序 MVP

1. 总体验收原则

本期验收以“微信小程序 MVP 是否跑通核心闭环”为第一标准。
展示网页不是本期核心验收对象。
后台管理端以“能承接和处理线索”为最低标准。

2. P0 验收范围（必须通过）

2.1 小程序首页与登录
验收项：
1）用户可进入首页
2）首页可进入需求录入和房源导入
3）需要保存或推进时，系统能引导登录
4）登录后可返回原流程

通过标准：
以上 4 项全部可用

2.2 需求录入
验收项：
1）用户可输入自然语言需求
2）系统可生成需求摘要
3）用户可修改并保存
4）保存后形成有效需求对象

通过标准：
可成功生成至少 1 份 submitted 状态需求

2.3 房源导入
验收项：
1）至少支持两种导入方式
2）导入后生成标准化房源
3）原始内容可保留
4）失败时可提示原因

通过标准：
至少 1 次成功导入 + 1 次失败提示正确

2.4 候选房源与详情
验收项：
1）系统可展示候选房源列表
2）可进行基础筛选
3）详情页能展示原始内容、结构化结果和缺失字段
4）可加入比较

通过标准：
候选列表、详情页、加入比较均可正常操作

2.5 房源比较
验收项：
1）支持 2~5 套房源比较
2）展示差异矩阵
3）生成推荐结论
4）支持保存比较

通过标准：
能完整完成一次 2 套房源比较，并生成 comparison_reports

2.6 风险确认
验收项：
1）比较后可进入风险确认
2）可看到总体风险等级
3）可看到风险标签和解释
4）高风险时能出现人工复核建议

通过标准：
能完整生成 1 条 risk_checks 记录

2.7 推进下一步
验收项：
1）可发起咨询中介
2）可发起预约带看
3）可请求人工复核
4）可稍后继续
5）至少一种动作后台可承接

通过标准：
至少 consult 和 appointment 两种动作可提交并在后台看见

2.8 我的空间
验收项：
1）用户可看到自己的需求
2）可看到自己的房源
3）可看到自己的比较
4）可看到自己的动作记录
5）可继续上次进度

通过标准：
至少 1 个用户完整数据链路可查看

2.9 后台管理端
验收项：
1）能查看线索列表
2）能查看线索详情
3）能更新状态
4）能查看关键日志

通过标准：
顾问可处理至少 1 条真实测试线索

3. 数据验收

必须满足：
1）buyer_intakes 有数据
2）listings 有数据
3）comparison_reports 有数据
4）risk_checks 有数据
5）next_actions 有数据
6）advisor_leads 或 appointments 至少其一有数据
7）activity_logs 有关键操作数据

4. 业务规则验收

必须验证：
1）比较对象少于 2 套不能进入正式比较
2）删除房源不能参与新比较
3）高风险时必须出现人工复核建议
4）未登录用户不能提交正式动作
5）动作提交后状态可追踪

5. 埋点验收

至少验证以下事件有记录：
1）需求提交
2）房源导入成功
3）房源导入失败
4）加入比较
5）生成比较
6）进入风险确认
7）点击咨询
8）点击预约
9）点击人工复核
10）保存稍后继续

6. 非功能验收（首期最低要求）

1）核心页面无明显白屏或阻断
2）关键失败有错误提示
3）导入失败不丢原始输入
4）动作失败有失败记录
5）小程序重新进入后能恢复最近状态或从我的空间继续

7. 通过标准总结

首期 MVP 通过的最小条件是：
用户可在微信小程序中完成
需求提交 -> 房源导入 -> 房源比较 -> 风险确认 -> 发起下一步
并且后台可承接线索，关键数据可追溯。

如果上述链路不能在小程序中独立完成，则本期视为未完成。

=== 2026-03 视觉与交互设计方针统一说明 ===

1. 整体基调：
- 风格：清透、专业、可信，以首页现有蓝灰到米白渐变为全站统一视觉方向。
- 交互主题：“缓”。买房是一个缓慢挑选、从容决策的过程，动效设计和交互反馈需保持平滑、柔和（如页面缓缓展开、面板平滑升起），避免急躁的视觉突变打断用户思考。
- 目的：降低高客单价交易的压迫感，强化“信息清晰、决策从容”的体验。

2. 核心色彩：
- 主背景层：低饱和蓝灰到米白渐变（#D6E0EF -> #DDE6F2 -> #F8F3EA），用于全站页面背景。
- 卡片/容器：白色或近白（#FFFFFF / rgba(255,255,255,0.96)），配合蓝灰描边。
- 主要强调色/按钮：橙色系（#F06A23 / #FF651C），用于主按钮、价格高亮、关键 CTA。
- 次级点缀色/标签：蓝灰色（#D6E0EF），用于标签、筛选、弱按钮和补充信息块。
- 文本颜色：#243247（主文本），#6B7A8F（次文本）。

3. 布局与组件约束：
- 导航架构：采用全局底部自定义 TabBar，包含“首页（左）、AI主入口（中，悬浮凸起）、我的（右）”。
- 卡片设计 (Card)：全站内容统一白卡 + 蓝灰描边结构，圆角统一 24rpx，阴影偏冷灰蓝（如 rgba(102, 123, 156, 0.12)）。
- 安全区：所有信息流页底部保留 180rpx 以上 padding，配合 env(safe-area-inset-bottom) 防遮挡。

4. 页面风格对齐要求：
- 首页作为全站视觉基准；其蓝灰渐变背景、白卡容器、橙色强调和轻量阴影应向登录、录入、导入、比较、风险、我的等页面统一。
- 不再新增木质/奶油色主视觉页面；如局部已有暖色资产，仅允许作为极小面积辅助色，不得主导页面观感。

8. 2026-03-24 补充：AI 最小决策闭环验收

验收项：
1）AI 页可从当前会话启动 1 个 decision_session。
2）decisionEngine start 成功后，decision_sessions 可写入 decision_session_id / state_json / result_json。
3）AI 页可展示三档候选（stable / balanced / value）中的至少一档结果。
4）完成 1 次 pairwise 后，state_json.soft_weights 或 pairwise_memory 有变化，且 state 接口返回的新候选结果可见变化。
5）提交 1 次 critique 后，state_json.critique_memory 或 blockers 有变化，且 state 接口仍可正常返回。
6）当可行候选不足时，relax 接口可返回至少 1 条放宽建议。
7）原 AI 聊天、会话摘要、memory_patch、收藏房源送 AI 分析链路不被破坏。

通过标准：
- `node --check` 通过：pages/ai/index.js、utils/decision.js、cloudfunctions/decisionEngine/index.js、process/decision_engine_smoke.js、cloudfunctions/queryPropertyRecommend/providers/openaiCompatible.js
- `node process/decision_engine_smoke.js` 通过
- 使用临时 apikey 执行 `node process/ai_provider_smoke.js` 时，openai_compatible 能返回兼容结构，且日志中不出现明文 key
