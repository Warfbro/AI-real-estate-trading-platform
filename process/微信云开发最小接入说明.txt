五、最小权限建议
1）users / buyer_intakes / listing_import_jobs / listings 应限制为登录用户按 user_id 读写自己的数据。
2）users 现已改为云函数服务端建档，避免客户端直写 users 被权限规则拦截。
3）buyer_intakes / listing_import_jobs / listings 目前仍有前端直写，正式环境必须配置数据库权限规则，不能只依赖页面层校验。
4）如果后续要把后台线索、动作中心正式上云，建议再把关键写操作迁到云函数。

六、为什么本次仍保留一个云函数
虽然当前目标是尽量少上传函数，但 `getOpenId` 仍然是必要的。
原因：
1）现在代码里的 `wx.login code` 不能作为正式稳定用户 ID。
2）如果没有 openid，用户数据写到云端后无法可靠归属到同一个微信用户。
3）`users` 在真机环境中可能被客户端权限规则拦截，登录建档需要服务端兜底。
4）买房用户快捷登录需要服务端结合 `phone_code` 解密手机号，前端不能把该逻辑当作本地假弹窗处理。
5）这会直接影响上线后正常运行，因此 `getOpenId` 属于必须保留的最小云函数。

七、验证方式
1）打开开发者工具并确保云开发环境可用。
2）重新部署 `getOpenId` 云函数后，以买房用户身份进入登录页，点击微信快捷登录，确认出现微信官方授权弹窗。
3）快捷登录成功后，检查 `users` 集合是否生成当前用户记录；若授权手机号，确认 `users.mobile` 已同步。
4）提交一次需求，检查 buyer_intakes 是否新增记录。
5）导入一次房源，检查 listing_import_jobs 和 listings 是否新增记录。
6）用截图导入一次，检查云存储是否生成文件，listing_import_jobs.raw_file_url 是否有 fileID。

八、当前限制
1）当前页面读取仍以本地数据为主，不是完整云真源模式。
2）如果云端同步失败，页面主流程仍会按本地数据继续执行。
3）要真正上线为“云端真源”，下一步需要继续改 compare / risk / action / admin 数据层。

十、AI 推荐工作流接入补充（2026-03-18）
1）新增云函数：`queryPropertyRecommend`
- 作用：由小程序 AI 页统一调用，服务端代理云端托管智能体接口。
- 入参：`query`、`user_id`、`session_id`、`source`、`context`。
- 出参：兼容智能体文档结构（`success/type/data/error/meta`），并补充 `trace_id` 与 `duration_ms`。

2）配置要求
- 推荐在云函数环境变量配置：`CLOUD_MANAGED_AGENT_ENDPOINT_URL`（或 `AI_MANAGED_AGENT_ENDPOINT_URL`）。
- 如需鉴权，配置：`CLOUD_MANAGED_AGENT_TOKEN`（或 `AI_MANAGED_AGENT_TOKEN`），云函数会自动追加 `Authorization: Bearer <token>`。
- 为兼容旧版本，若上述地址未配置，会回退读取 `PROPERTY_RECOMMEND_WEBHOOK_URL`。
- 未配置任何地址时云函数返回 `AGENT_ENDPOINT_MISSING`，不会中断小程序运行。

3）日志与追踪
- 云函数会写 `activity_logs`：
  `action_type=ai_property_recommend_query`，
  `object_type=ai_recommendation`，
  并记录 `trace_id/session_id/source/query_length/success/error_code/duration_ms/engine_mode`。
- 不落用户原始 query 明文，避免敏感信息泄漏。

4）前端接入点
- `utils/cloud.js` 新增 `queryPropertyRecommend` 封装。
- `pages/ai/index` 新增：
  输入发送、推荐结果渲染、澄清问题渲染、错误提示、AI 上下文摘要与主建议动作。

5）部署与验证补充
- 需要在云开发中部署新增云函数 `queryPropertyRecommend`。

十二、2026-04-07 补充：agent HTTP 接口暴露

1）新增云函数
- `agentHttpGateway`

2）新增路由清单与部署脚本
- 路由清单：`process/agent_http_routes_manifest.json`
- 部署脚本：`process/deploy_agent_http_gateway.ps1`

3）需要暴露的 HTTP 路径
- `/agent/query-user-state`
- `/agent/query-houses-brief`
- `/agent/query-houses-detail-by-ids`

4）调用约束
- 方法：`GET` / `POST`
- Header：`Authorization: Bearer <token>`
- 云函数环境变量：`AGENT_HTTP_API_TOKEN`
- 兼容别名环境变量：`CLOUD_AGENT_HTTP_API_TOKEN`

5）部署步骤
- 先确保本机已完成 CloudBase CLI 登录
  - 全局已装 CLI 时可执行：`tcb login`
  - 未全局安装时可执行：`npx --yes --package=@cloudbase/cli tcb login`
- 先为 `agentHttpGateway` 配置 bearer token 环境变量
- 再执行：
  - `powershell -ExecutionPolicy Bypass -File process/deploy_agent_http_gateway.ps1 -EnvId <your-env-id>`
- 若需要先清理旧路由再重建：
  - `powershell -ExecutionPolicy Bypass -File process/deploy_agent_http_gateway.ps1 -EnvId <your-env-id> -ResetRoutes`
- 脚本优先调用本机 `tcb`；若未安装，会自动回退到 `npx --yes --package=@cloudbase/cli tcb`

6）脚本实际执行内容
- `tcb fn deploy --dir cloudfunctions/agentHttpGateway --force -e <env-id>`
- `tcb service create -p /agent/query-user-state -f agentHttpGateway -e <env-id>`
- `tcb service create -p /agent/query-houses-brief -f agentHttpGateway -e <env-id>`
- `tcb service create -p /agent/query-houses-detail-by-ids -f agentHttpGateway -e <env-id>`

7）最小验证步骤
- 用正确 bearer token 调 `GET /agent/query-user-state?uid=<user_id>`，预期返回 `200` 且 `data.user_preference` / `data.allowed_ai_state` 存在
- 调 `POST /agent/query-houses-brief`，body 至少含 `uid` 与 `mode=brief`，预期返回 `items` 且每条含 `isFavorited`
- 调 `POST /agent/query-houses-detail-by-ids`，body 含 `uid` 与 `houseIds`，预期返回 `mode=detail`
- 不带 token 或 token 错误时，预期返回 `401 UNAUTHORIZED`

8）日志观察点
- `activity_logs`
- `action_type=agent_http_port_query`
- 重点检查：`trace_id`、`path`、`status_code`、`duration_ms`

9）回退方式
- 先删除对应 HTTP service 路由，再回退云函数目录 `cloudfunctions/agentHttpGateway`
- 若只需停用外部入口，可保留本地 `modules/aiAssistant/agentPorts.js` 不动
