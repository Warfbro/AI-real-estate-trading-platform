完整路线不是“先写聊天，再慢慢补算法”，而是反过来：先把可计算对象定出来，再做学习、求解、解释，最后才是对话包装。
因为会话式推荐的重点本来就不是单轮应答，而是 richer interactions、偏好获取与反馈循环；复杂商品更适合显式规则/约束；住房选择本身又是典型的多准则 trade-off 问题；而 pairwise 查询之所以常见，是因为认知负担低，适合实时偏好获取。

下面这条路，才是你这个项目从“现在的壳子”走到“正式系统”的完整完成路径。

0. 先定最终数学对象

在动手前，先统一 4 个对象，不然后面所有代码都会漂。

第一，房源特征 phi(x)
每套房要被映射成固定长度向量，比如：

学区稳定性
通勤成本
房龄/维护风险
小区环境
流动性
总价与月供压力
税费/交易成本
周边配套

第二，用户效用函数 u(x)
第一版正式版直接用：
u(x) = w^T phi(x)
其中 w 不是手写权重，而是系统逐步学出来的用户偏好参数。

第三，约束集合 C(x)
比如预算、电梯、通勤上限、最晚入学时间，这些决定可行域。约束型推荐最适合处理这种显式规则与商品属性映射。

第四，解释对象
解释不是文案，而是 4 类结构化信息：

证据
贡献
反事实
不确定性

只要这 4 个对象定住，后面所有模块就不会散。

1. 第一阶段：把“数据底座”做出来

这一步不高级，但是后面一切算法的前提。

你要先完成 5 件事：

建房源主表
至少要有：
基础属性
交易属性
学区属性
通勤属性
风险属性
做房源特征工程
把每套房转成 phi(x)。
这里不要一开始就追求“全自动 embedding”，而是先做结构化、可解释特征。
建决策会话表
一条买房决策不是一条聊天记录，而是一条长期 session。
建偏好状态表
不是只存“聊天摘要”，而是要存：
hard_constraints
w_mean
w_var
evidence_log
interaction_state
建交互事件表
每次用户的：
pairwise 选择
critique
候选点击
比较
都必须能回放。
这一阶段的完成标准

你可以做到：

不接 LLM，也能把一套房表示成稳定特征
不跑推荐，也能把用户每次行为记成标准事件
不做解释，也能把偏好状态落库
2. 第二阶段：先做“可行域”，不是先做对话

这一阶段只解决一个问题：

在用户约束下，哪些房子是可行的？

先别急着学偏好，也别急着解释。

做法

先实现一个 Feasibility Engine：

输入：

hard_constraints
house_pool

输出：

feasible_set
第一版实现

如果只是单套房筛选，先用规则过滤就够。
比如：

price <= budget_max
monthly_payment <= payment_max
has_elevator == True
commute_minutes <= max_commute
为什么先做这一步

因为后面不管你偏好学得多好，不可行的东西都不该进候选。
这部分本质属于约束满足问题，复杂后可以升级到 CP-SAT / MILP 这类求解器；Google OR-Tools 官方就把 CP-SAT 定位为主要的约束编程求解器，适合整数约束与组合优化建模。

这一阶段的完成标准

给一组约束，系统能稳定输出：

可行候选数
被过滤原因
哪个约束杀伤最大
3. 第三阶段：把“偏好状态”升级成后验，不要再写死权重

这一步才是系统真正长技术内核的时候。

目标不是“用户喜欢学校，权重 0.8”，而是：

系统维护用户偏好参数 w 的后验状态。

推荐实现

第一版正式版直接做：

w_mean: 每维偏好均值
w_var: 每维偏好方差
evidence_log: 每次更新依据

这比存一个静态 soft_preferences 强很多。

更新来源

只允许三类来源更新偏好：

pairwise 选择
例如：
A 比 B 更适合我
我更不能接受学校一般，而不是月供更紧
critique
例如：
太老了
学校不稳
月供太顶
明确表达
例如：
我就是学区优先
我不接受没电梯
更新方法

推荐你第一版就上 pairwise Bayesian update，不要再用“LLM 直接改权重”。

成对选择可以建模成：
P(A > B) = sigmoid(u(A) - u(B))

然后对 w_mean, w_var 做近似在线更新。
Guo 和 Sanner 这条路线就是做多属性 Bayesian preference elicitation，并且强调 pairwise 查询低认知负担，适合实时更新效用信念状态。

这一阶段的完成标准

你能做到：

用户做一次 pairwise，w_mean 会变化
用户吐槽一次，系统能把 critique 转成偏好证据
系统能区分“高置信偏好”和“还没学稳的偏好”
4. 第四阶段：把“下一问什么”做成主动学习，不再补槽位

这是从“表单机器人”跨到“正式系统”的关键一步。

目标不是问缺什么，而是：

问哪一个问题，最能减少推荐错误，同时不让用户烦。

推荐做法

做一个 Query Selector。

输入：

当前 w_mean, w_var
当前 top 候选集
当前交互状态
当前 critique

输出：

下一问的 query
query 类型优先级

第一优先：真实候选之间的 pairwise
第二优先：边界探测
第三优先：critique 反演
最后才是开放式追问。

query 打分

不要只看信息增益。
正式版建议用：

Score = 预计后悔降低 + 信息增益 - 认知负担 - 疲劳惩罚

因为房产是高风险、低频决策，核心不是“学到更多”，而是“尽快别推错”。
主动偏好获取和实时 Bayesian elicitation 的核心本来就是 query selection，而不是随便追问。

这一阶段的完成标准

你能做到：

系统不是“没预算就问预算”
系统会围绕当前最关键 trade-off 发问
两个不同用户，即使字段都缺，下一问也可能不一样
5. 第五阶段：把决策引擎升级成“效用 + 风险 + 分桶”

这一步才是真正把“会选”做出来。

目标

不是给一个总分排序，而是给出：

哪些房子在当前偏好下最优
哪些更稳妥
哪些更有性价比但更冒险
推荐做法

在可行集 F 上，对每套房计算：

ExpectedUtility = phi(x)^T * w_mean
Uncertainty = phi(x)^2 * w_var

于是你就同时有：

期望值
风险/不确定度
输出方式

不要只给 Top-3。
固定分 3 桶最合理：

稳妥型：高期望、低方差
均衡型：综合最好
性价比型：某些维度牺牲一点，但价格更友好

住房多准则 DSS 的关键就在于：要表达 trade-off，而不是 pretending “只有一个绝对最优”。

这一阶段的完成标准

你能做到：

候选不是一串总分
系统能明确区分“稳妥”和“更值”
排序是由当前偏好后验驱动，不是 prompt 临场发挥
6. 第六阶段：把“放宽建议”做成真正的求解问题

这是最适合引入求解器的阶段。

当候选为空或很少时，真正要解的是：

放宽哪一个条件、放宽多少，收益最大、代价最小。

正确做法

先定义放宽动作集：

预算 +5 万 / +10 万
通勤 +10 分钟 / +20 分钟
电梯从硬约束改强偏好
房龄从硬要求改软要求

然后优化：

Gain(a) - lambda * Cost(a)

其中：

Gain(a) = 可行候选增加多少，或 top 候选效用提升多少
Cost(a) = 用户语义上放宽的痛感
什么时候上求解器

如果放宽动作少，先枚举也行。
如果后面你做成：

多动作组合
多阶段计划
置换链
时间窗

就直接上 OR-Tools CP-SAT。
OR-Tools 官方文档明确说 CP-SAT 是其主要约束编程求解器，适合整数约束问题，并返回 OPTIMAL / FEASIBLE / INFEASIBLE 等状态，很适合你这种“放宽路径求解”。

这一阶段的完成标准

你能做到：

不再用硬编码 if/else 给放宽建议
能算出“最划算的放宽动作”
同一条件下，不同用户的放宽建议可能不同
7. 第七阶段：解释层升级成“证据 + 贡献 + 反事实”

这一步不是最后补文案，而是把系统真正做成“顾问”。

正确目标

解释层至少要回答 4 件事：

我为什么这样理解你
这套房为什么赢
如果你改一个取舍，结果会怎么变
现在哪些地方我还没完全学稳
实现方式

先做 Explanation Planner，输出结构化解释计划：

evidence_summary
contribution_breakdown
counterfactuals
uncertainty_summary
next_step_type

再让 LLM 做 Surface Realization。

关键点

LLM 不负责“想理由”。
LLM 只负责把 planner 给出的理由说自然。

这一阶段的完成标准

你能做到：

系统解释有证据出处
能说“为什么不是 B”
能说“如果你放松月供约束，B 会反超 A”
用户能感觉到系统不是在乱编
8. 第八阶段：把“自然聊天”接回去

这一步才轮到你现在最熟的那部分。

目标是：

把已经成熟的学习/求解内核，包成一个低打扰、有情绪价值的顾问式交互。

正确分工
LLM 负责：
情绪承接
自然语言理解
critique 转结构化线索
话术包装
算法层负责：
状态更新
query selection
decision
relaxation
explanation planning
到这里，系统才算成型

否则就是“先做聊天，后面补算法”，那大概率永远补不稳。

9. 第九阶段：离线验证，不要直接线上盲试

上线前你至少要做 4 类离线验证：

偏好恢复测试
用模拟用户或历史案例，看系统能不能在少量 pairwise 后逼近正确偏好。
候选质量测试
给定固定约束和固定偏好，看 top 候选是否稳定。
放宽建议质量测试
检查放宽建议是否真的提升候选数量或效用，而不是乱放。
解释一致性测试
检查解释是否和真实 solver trace / utility breakdown 一致。
10. 第十阶段：上线顺序

真正的上线顺序我建议这样：

第 1 版
可行域过滤
静态 w_mean
基础 compare
基础解释
第 2 版
pairwise 更新
critique 更新
query selector
三桶输出
第 3 版
放宽求解
反事实解释
不确定性解释
第 4 版
多步方案
置换链
行动规划
更强求解器接入
11. 最终一句话版完成路径

完整完成路径就是：

先把房源表示成特征，再把用户偏好表示成后验状态；
再用主动偏好获取去学用户的效用函数；
再在约束可行域里做期望效用与风险决策；
再把无解问题交给放宽求解器；
最后用证据驱动的解释层把结果说成人话。

这条路里：

学习的部分：学用户偏好
求解器的部分：解可行域、放宽路径、后续组合规划
对话的部分：只做接口，不做真相源

这才是完整路径。
