AI买房决策顾问_架构文档

一、总体架构结论
本系统不是纯知识库问答机器人，也不是纯提示词工程。
正式实现采用：
“陪伴式对话外壳 + 显式偏好状态 + 低打扰主动偏好获取 + 约束/多准则决策引擎 + 顾问式解释层”的混合架构。

统一链路为：
小程序前端
  ->
API Gateway / Session Gateway
  ->
Dialogue Orchestrator
  ├─ LLM 对话层
  ├─ Emotion & Intent Extractor
  ├─ Preference State Service
  ├─ Question Selector
  ├─ Decision Engine
  ├─ Explanation Generator
  └─ RAG / Rule Service
  ->
业务数据库 / 房源库 / 学区库 / 通勤库 / 日志库

二、分层设计
1. 前端层
职责：
（1）聊天UI
（2）候选房源卡片
（3）房源对比展示
（4）收藏、咨询、预约等动作入口

设计原则：
（1）前端不直连模型
（2）前端不直连Coze等平台
（3）统一走自有后端

2. API Gateway / Session Gateway
职责：
（1）用户身份识别
（2）会话路由
（3）请求鉴权
（4）限流与成本控制
（5）日志埋点入口

3. Dialogue Orchestrator
职责：
（1）统一编排整轮对话
（2）调度LLM、偏好状态服务、问题选择器、决策引擎、解释层
（3）控制每轮输出形态
（4）保证“承接-判断-推进”的回复结构

说明：
这层可以临时借助Coze做编排，但不能让它成为系统唯一真相源。

4. 陪伴式对话层
职责：
（1）理解用户自然语言
（2）识别情绪和语气
（3）在自然对话中承接焦虑
（4）把内部计算结果说成人话
（5）把需要问的问题包装成低打扰问法

边界：
（1）不负责最终排序
（2）不负责约束冲突求解
（3）不负责业务真相存储

5. 偏好状态层
职责：
（1）维护硬约束
（2）维护软偏好
（3）维护置信度
（4）维护不确定项
（5）维护critique记忆
（6）维护pairwise记忆
（7）维护交互状态

设计原则：
偏好状态层是系统真相源。
不能只把用户需求放在聊天上下文里。

6. 低打扰问题选择器
职责：
（1）决定当前是否该问
（2）决定下一问问什么
（3）决定问题应该怎么问

目标：
不是把字段问全，而是用最少打扰换最多有效偏好信息。

策略优先级：
（1）pairwise二选一
（2）critique引导
（3）范围确认
（4）开放式追问

7. 决策引擎
职责：
（1）可行性检查
（2）硬约束过滤
（3）软偏好打分
（4）候选生成
（5）多样性控制
（6）冲突检测
（7）放宽建议

说明：
住房选择是多准则、高风险决策。
因此真正的排序和可行性判断必须由计算内核承担，不应由LLM临场拍板。

8. 顾问式解释层
职责：
（1）解释为什么推荐这几套
（2）解释做了哪些取舍
（3）解释为什么要放宽某些条件
（4）输出用户能接受的话术

解释要求：
（1）不报表
（2）不念分数
（3）要体现“我帮你理过了”

9. 知识与规则层
职责：
（1）政策问答
（2）学区规则
（3）税费规则
（4）贷款规则
（5）风险规则
（6）知识补充检索

边界：
RAG/规则层负责补知识，不负责核心排序。

三、核心数据流
1. 首轮进入
（1）用户输入自然语言
（2）对话层识别情绪、意图、偏好线索
（3）偏好状态层初始化状态
（4）决策引擎基于已知条件生成粗候选
（5）问题选择器判断先给候选还是先轻问一轮
（6）解释层输出顾问式回复

2. 用户反馈/吐槽
（1）用户对候选做自然语言反馈
（2）critique parser转换为偏好更新
（3）偏好状态更新
（4）决策引擎重排候选
（5）问题选择器决定下一轮推进点
（6）解释层输出新结果

3. 无解流程
（1）决策引擎发现可行集为空或极小
（2）Conflict Detector输出冲突组合
（3）Relaxation Generator计算最小代价放宽路径
（4）解释层将其转成顾问式建议

四、服务边界
1. 前端负责
（1）聊天展示
（2）候选卡片
（3）对比展示
（4）收藏与行动按钮

2. 自有后端负责
（1）用户身份
（2）会话状态
（3）偏好状态
（4）决策引擎
（5）规则与数据访问
（6）日志与监控
（7）成本控制

3. LLM / Coze 类编排层负责
（1）对话理解
（2）情绪承接
（3）自然语言抽取
（4）文本解释

不负责：
（1）真实业务真相源
（2）真实候选排序真相源
（3）用户核心数据长期存储

五、部署建议
推荐组件：
（1）小程序前端
（2）业务API服务
（3）偏好状态服务
（4）决策引擎服务
（5）检索/规则服务
（6）MySQL 或 PostgreSQL
（7）Redis
（8）向量库（仅用于知识检索）
（9）LLM服务（可接Coze / 自有模型接口）

部署原则：
（1）小程序不直连LLM平台
（2）决策逻辑不写死在prompt
（3）偏好状态入库
（4）每次推荐结果可回溯

六、架构设计原则
1. 体验优先但不牺牲计算内核
对外要像真人顾问，对内必须是显式状态和决策引擎。

2. LLM只做语言接口
LLM负责理解、承接、解释，不负责核心排序和冲突求解。

3. 问题选择器同时优化信息增益和打扰成本
不能只问得准，还要问得顺。

4. 没有完美解时给放宽建议
不能把“无解”当失败，而要把它变成取舍引导。

5. 状态必须独立于prompt长期存在
否则后续必然退化成堆提示词和上下文补丁。

七、分阶段演进
Phase 1：MVP
（1）显式偏好状态
（2）简单规则过滤
（3）简单加权排序
（4）顾问式解释模板

Phase 2：正式版
（1）交互状态
（2）低打扰问题选择器
（3）pairwise偏好获取
（4）冲突检测
（5）放宽建议
（6）多方案输出

Phase 3：高级版
（1）主动偏好学习
（2）更强的效用学习
（3）高维偏好建模
（4）更强的优化求解内核
